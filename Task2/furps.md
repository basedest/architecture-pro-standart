# FURPS+ — архитектурно-значимые требования. Цифровизация депозитного и кредитного направлений банка «Стандарт»

| Код | Требования | Комментарий |
|-----|------------|-------------|
| **F** | **Функциональные (Functionality)** | |
| F1 | Сайт: новый клиент видит список доступных депозитов с актуальными ставками. | Ставки должны публиковаться на сайт из внутреннего источника (сейчас ведутся только в XLS-файлах — см. F7). |
| F2 | Сайт: клиент подаёт заявку на депозит, указав Ф. И. О. и номер телефона. Заявка передаётся в систему кол-центра. | После подачи заявки клиенту звонит менеджер кол-центра. |
| F3 | Система кол-центра: менеджер видит заявку с сайта, изучает её и может предложить клиенту особые условия по депозиту. | Особые условия согласуются по текущему процессу через бэк-офис. |
| F4 | Интернет-банк: клиент видит список доступных депозитов с актуальными ставками и персонализированные ставки лично для него. | Персонализированные ставки рассчитываются по текущему процессу согласования (бэк-офис депозитов + отдел кредитования). |
| F5 | Интернет-банк: клиент подаёт заявку на открытие депозита, указав счёт и сумму. Операция подтверждается СМС-кодом. | Функционал СМС в ядре платформы ИБ требует доработки подрядчиком — этого нужно избежать; бизнес-функционал СМС-подтверждения реализуется силами команды банка (см. +R2). |
| F6 | MVP: заявку на открытие депозита обрабатывает менеджер бэк-офиса и подтверждает условия депозита в АБС. | Полная автоматизация без участия бэк-офиса — цель на год, вне MVP. |
| F7 | Централизованное ведение ставок по депозитам (сейчас — только XLS-файлы). Со ставками работают сотрудники бэк-офиса депозитов и бэк-офиса кредитов. Функциональность реализуется в АБС, как и текущий процесс согласования ставки по заявке. | Единый источник ставок нужен сайту (F1), интернет-банку (F4) и кол-центру. |
| F8 | СМС-уведомления клиенту: после подтверждения размера ставки и после открытия депозита. | Отправка через существующий СМС-шлюз телеком-оператора. |
| F9 | Безопасность: чувствительные данные при передаче с сайта и в интернет-банке защищаются шифрованием трафика (TLS/HTTPS). | Безопасность относится к группе F в модели FURPS. |
| F10 | Безопасность: сотрудники депозитного и кредитного направлений не должны иметь доступа к данным друг друга. | Действующее требование безопасности банка; влияет на модель доступа к ставкам в АБС (F7). |
| F11 | Сайт: новый клиент видит предложение открыть кредит по ставке; полная информация об условиях кредита хранится в Кредитном конвейере. Клиент подаёт заявку, указав Ф. И. О. и номер телефона. | Условия кредитов должны публиковаться на сайт из Кредитного конвейера. |
| F12 | Сайт: если клиент дополнительно заполнил паспортные данные, ему сразу выдаётся решение об одобрении/неодобрении кредита на основании данных системы скоринга. Для нового клиента используются только общие данные бюро кредитных историй. | Онлайн-вызов скоринга в момент подачи заявки; данных АБС о новом клиенте нет. |
| F13 | Сайт: после подачи заявки на кредит клиент получает СМС о необходимости прийти в отделение для идентификации и получения кредита. | |
| F14 | Отделение: менеджер подаёт заявку на кредит через АБС. Если у клиента уже есть предварительное одобрение с сайта, сотрудник продолжает работу с той же заявкой; для полностью нового клиента создаётся новая заявка. | Требуется сквозная идентификация заявки между сайтом, Кредитным конвейером и АБС. |
| F15 | Интернет-банк: клиент видит список доступных кредитов с актуальными условиями, а также предодобренное предложение по кредиту (если по клиенту ранее рассчитан скоринг и приемлемая сумма). | |
| F16 | Интернет-банк: клиент подаёт заявку на кредит, указав счёт зачисления средств и данные заявки. Операция подтверждается СМС-кодом. После подачи заявки по предодобренному предложению скоринг выполняется повторно по упрощённой процедуре, если ситуация изменилась. | Такие заявки сотрудники обрабатывают быстрее. |
| F17 | СМС-информирование клиента об изменении статуса заявки на кредит. | |
| **U** | **Удобство использования (Usability)** | |
| U1 | Интерфейс работы должен быть максимально удобным для клиента. | Ожидание бизнеса. |
| U2 | Приложения соблюдают принятую в компании систему дизайна: привычные цвета и брендированные элементы. | Относится к сайту и интернет-банку. |
| **R** | **Надёжность (Reliability)** | |
| R1 | Все сервисы работают 24/7 и доступны в 99,9% случаев. | Относится и к новым сервисам депозитов/кредитов, и к интернет-банку. |
| R2 | При сбое в основном ЦОД сервисы интернет-банка остаются доступны и выдерживают требуемую нагрузку (переключение на резервный ЦОД). | Команда ИБ считает требование пока невыполнимым: всё работает в одном ЦОД на одной группе серверов. Требование фиксируется как целевое — оно архитектурно значимо для решения. |
| R3 | Исключить прямую работу интернет-банка с API/БД АБС в новом процессе: БД АБС перегружена, онлайн-поток заявок не должен ставить под угрозу работоспособность банка. | Нужна асинхронная развязка (очередь/буфер) между онлайн-каналами и АБС. |
| R4 | Гарантированная доставка заявок и СМС-событий между системами: заявка, принятая онлайн-каналом, не должна теряться при сбое смежной системы. | Дополнительное требование: при асинхронной интеграции (R3) без гарантий доставки заявки будут теряться молча, что напрямую бьёт по выручке и доверию клиентов. |
| **P** | **Производительность (Performance)** | |
| P1 | Отклик по всем клиентским операциям — максимально быстрый, в пределах миллисекунд. Исправить текущую проблему: при проведении платежей справочные данные загружаются более секунды. | Требует кеширования справочных данных/ставок на стороне онлайн-каналов. |
| P2 | Равномерное горизонтальное масштабирование и распределение запросов между серверами, приложениями и ЦОД. | АБС масштабируется только вертикально из-за своей БД (см. +R4) — горизонтально масштабируются онлайн-каналы и новые сервисы. |
| P3 | Заявки на кредит по предодобренным предложениям обрабатываются сотрудниками в течение того же рабочего дня. | Сейчас заявки попадают в Кредитный конвейер раз в сутки из АБС; ускорить обмен между базами невозможно (+R3) — нужен отдельный онлайн-канал доставки заявок в Кредитный конвейер. |
| P4 | Систему кредитного скоринга нельзя нагружать дополнительными процессами предрасчёта скорингов: она критична для банка и испытывает повышенную нагрузку в рабочие часы. | Предодобренные предложения рассчитываются вне пиковых часов либо на отдельных мощностях. |
| **S** | **Поддерживаемость (Supportability)** | |
| S1 | При доработках максимально использовать технологии, уже существующие в банке (СУБД MS SQL и Oracle и др.). Новые технологии допустимы, если совместимы с существующими платформами разработки и по ним есть экспертиза внутри банка хотя бы на уровне языков программирования. | Доступный стек: .NET, Delphi/PL-SQL, Java (Spring Boot, Camunda), Python/Flask, PHP/React.js, PostgreSQL. |
| S2 | Если нужны очереди сообщений — использовать Kafka (на перспективу). | Текущая платформа интернет-банка (ASP.NET MVC 4.5 / .NET Framework 4.5) несовместима с Kafka (+R5) — интеграция через отдельный сервис/адаптер. |
| S3 | Рассмотреть перевод интернет-банка на микросервисную архитектуру — пока только в рамках задачи открытия депозитов. | Позволяет выносить новый функционал из монолита подрядчика и обходить ограничение +R5. |
| S4 | Разработать документацию для дальнейшего расширения системы. | |
| S5 | Мониторинг и журналирование операций по онлайн-заявкам (аудит: кто, когда, какая заявка, какой статус). | Дополнительное требование: банковские операции требуют аудиторского следа; без него невозможно разбирать инциденты по потерянным заявкам и СМС и подтверждать SLA из R1/P3. |
| **+R** | **+ Ограничения (Restrictions)** | |
| +R1 | Новые клиенты обязаны пройти личную идентификацию с проверкой документов в отделении банка. | Регуляторное ограничение: банки не могут принимать новых клиентов без личной проверки документов. |
| +R2 | Ядро платформы интернет-банка дорабатывается только подрядчиком; обновление ядра привязано к подрядчику. Доработок ядра следует избегать — бизнес-функционал реализуется силами команды банка. | Ограничивает варианты реализации СМС-подтверждения (F5) и интеграций. |
| +R3 | Обмен данными между АБС и Кредитным конвейером — интеграция по БД раз в сутки; ускорить обмен и проводить его чаще невозможно. | Ключевое ограничение для P3. |
| +R4 | АБС масштабируется только вертикально из-за своей БД; БД АБС уже перегружена. | Ключевое ограничение для R3 и P2. |
| +R5 | Текущая версия платформы интернет-банка (ASP.NET MVC 4.5 / .NET Framework 4.5, MS SQL) несовместима с Kafka. | Ограничение для S2. |
| +R6 | Система кол-центра — платформа подрядчика (React.js / Java Spring Boot / PostgreSQL, микросервисы); доработки выполняет команда сопровождения подрядчика (5 человек) либо Java-специалисты команды АБС. | Ограничивает объём и скорость доработок системы кол-центра (F3). |
| +R7 | Кредитный конвейер — коробочное решение на Camunda BPM (Oracle); скоринг — внутренняя разработка (Python/Flask, PostgreSQL); Бюро кредитных историй — внешняя система, доступ только через REST API. | Фиксирует технологические рамки кредитного процесса (F11–F16). Кредитный конвейер и скоринг поддерживают горизонтальное масштабирование и работу 24/7. |

### **Название задачи:** ADR-1. Концептуальная архитектура MVP открытия депозитов онлайн
### **Автор:** basedest
### **Дата:** 19.07.2026
### **Функциональные требования**

Цель MVP на горизонте 6 месяцев — дать клиенту возможность оставить заявку на депозит через онлайн-каналы (сайт и интернет-банк), при этом обработку заявки и подтверждение условий по-прежнему выполняет менеджер бэк-офиса в АБС (F6, полная автоматизация — цель года, вне MVP). Требования взяты из `Task2/furps.md` (коды F/U/R/P/S/+R). Ниже — верхнеуровневые Use Cases основного потока.

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
| UC1 | Новый клиент, Сайт, Сервис депозитов | Просмотр ставок на сайте (F1) | Сайт запрашивает актуальные ставки REST-вызовом у Сервиса депозитов (чтение из кеша — P1) и показывает список доступных депозитов. |
| UC2 | Новый клиент, Сайт, Сервис депозитов, Система КЦ, Оператор КЦ | Заявка на депозит с сайта (F2, F3) | Клиент оставляет Ф. И. О. и телефон (HTTPS — F9) → Сервис депозитов сохраняет заявку и создаёт обращение в системе КЦ (REST) → менеджер КЦ звонит и может предложить особые условия → клиент идёт в отделение для идентификации (+R1). |
| UC3 | Клиент, Интернет-банк, Сервис депозитов | Просмотр ставок и персональных ставок в ИБ (F4) | Модуль депозитов ИБ запрашивает актуальные ставки и персональные предложения у Сервиса депозитов (REST, чтение из кеша — P1). |
| UC4 | Клиент, Интернет-банк, Сервис депозитов, СМС-шлюз | Подача заявки на депозит в ИБ с СМС-кодом (F5, F9) | Клиент указывает счёт и сумму → Сервис депозитов генерирует ОТП-код и отправляет его через СМС-шлюз (бизнес-логика СМС вне ядра ИБ — +R2) → после подтверждения заявка публикуется в Kafka (R3, R4). |
| UC5 | Интеграционный адаптер АБС, АБС, Менеджер бэк-офиса депозитов | Обработка заявки в АБС (F6) | Адаптер читает заявку из Kafka и создаёт её в АБС с контролем темпа записи (+R4) → менеджер бэк-офиса обрабатывает заявку и подтверждает условия депозита в АБС. |
| UC6 | АБС, Интеграционный адаптер АБС, Сервис депозитов, СМС-шлюз, Клиент | СМС-уведомления о ставке и открытии депозита (F8) | АБС фиксирует статус (ставка подтверждена, депозит открыт) → адаптер публикует событие в Kafka → Сервис депозитов отправляет СМС клиенту через шлюз. |
| UC7 | Менеджер бэк-офиса депозитов, Менеджер бэк-офиса кредитов, АБС, Интеграционный адаптер АБС | Ведение ставок в АБС (F7, F10) | Сотрудники ведут ставки в новом модуле АБС с разграничением доступа депозиты/кредиты; адаптер публикует изменения ставок в Kafka для обновления кеша Сервиса депозитов. |

### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| R1 | Все сервисы работают 24/7 с доступностью 99,9% (новые сервисы депозитов и интернет-банк). |
| R2 | При сбое основного ЦОД сервисы ИБ остаются доступны (целевое требование; в MVP закрывается ручным переключением на резервный ЦОД). |
| R3 | Исключить прямую работу ИБ и онлайн-каналов с API/БД перегруженной АБС — асинхронная развязка через очередь. |
| R4 | Гарантированная доставка заявок и СМС-событий между системами; заявка не теряется при сбое смежной системы. |
| P1 | Отклик по клиентским операциям в пределах миллисекунд за счёт кеширования ставок на стороне онлайн-каналов/сервиса. |
| P2 | Горизонтальное масштабирование новых сервисов и онлайн-каналов; АБС масштабируется только вертикально (+R4). |
| S1 | Использовать технологии с внутренней экспертизой банка (Java Spring Boot, PostgreSQL, Oracle/PL-SQL). |
| S2 | Для очередей сообщений использовать Kafka. |
| S3 | Новый функционал депозитов выносится из монолита ИБ в микросервисы. |
| S4 | Документация для дальнейшего расширения системы. |
| S5 | Мониторинг и журналирование операций по онлайн-заявкам (аудит). |
| U1 | Максимально удобный интерфейс для клиента. |
| U2 | Соблюдение корпоративной системы дизайна на сайте и в ИБ. |
| +R1 | Новые клиенты проходят личную идентификацию в отделении. |
| +R2 | Ядро платформы ИБ дорабатывается только подрядчиком — доработок ядра избегаем, бизнес-логику пишет команда банка. |
| +R4 | АБС масштабируется только вертикально, БД АБС перегружена — контроль темпа записи заявок. |
| +R5 | Платформа ИБ (ASP.NET MVC 4.5 / .NET Framework 4.5) несовместима с Kafka — интеграция через отдельный сервис/адаптер. |

### **Решение**

**Диаграммы C4** (загружены отдельными файлами в директорию Task3):

- `c4-context-deposits.drawio` — уровень 1, контекст. Показывает клиентов и все системы, участвующие в открытии депозитов онлайн: Сайт, Интернет-банк, новый Сервис депозитов, Kafka, АБС (с новым модулем ставок), Систему КЦ, СМС-шлюз и телеком-оператора. Стрелки соответствуют Use Cases UC1–UC7: REST/HTTPS-запросы ставок, создание обращения в КЦ, асинхронный обмен заявками/ставками/событиями через Kafka, отправка СМС. Диаграмма подтверждает ключевой принцип R3 — онлайн-каналы не обращаются к АБС напрямую, весь поток идёт через Сервис депозитов и Kafka.
- `c4-container-deposits.drawio` — уровень 2, контейнеры. По требованию задания детализированы Интернет-банк и АБС. Внутри ИБ выделены неизменяемое ядро подрядчика (веб-приложение на ASP.NET MVC 4.5, БД MS SQL) и новый модуль депозитов (страницы и контроллеры, разрабатываются командой банка — +R2). Внутри АБС показаны Delphi-клиент бэк-офиса, БД Oracle с новым PL/SQL-модулем ставок и депозитных заявок, а также новый Интеграционный адаптер АБС на Java Spring Boot. Сервис депозитов раскрыт на контейнеры: REST API, БД PostgreSQL (заявки, кеш ставок, ОТП) и модуль СМС-уведомлений. Обозначены топики Kafka: `deposit-applications`, `deposit-rates`, `deposit-status-events`.

**Логика принятия решений** (каждый пункт обоснован кодами требований и указывает ответственную команду):

1. **Kafka как единая шина асинхронного обмена (S2).** Между онлайн-каналами и АБС ставится Kafka: она развязывает быстрый онлайн-поток заявок и перегруженную БД АБС (R3, +R4) и даёт гарантированную доставку через персистентные топики (R4). Разворачивает и сопровождает — команда инфраструктуры/IT.
2. **Сервис депозитов — новый микросервис (Java Spring Boot + PostgreSQL).** Стек выбран по S1: Java-экспертиза уже есть в команде АБС, тот же стек у системы КЦ и Camunda; функционал выносится из монолита ИБ (S3). Вся Kafka-интеграция вне монолита ИБ, т. к. платформа ИБ несовместима с Kafka (+R5). Разрабатывают Java-специалисты команды АБС.
3. **Интеграционный адаптер АБС — новый контейнер (Java Spring Boot) рядом с АБС.** Потребляет заявки из топика `deposit-applications` и создаёт их в АБС с контролем темпа записи (+R4); публикует в Kafka справочник ставок (`deposit-rates`) и события статусов (`deposit-status-events`). ИБ, сайт и Сервис депозитов не работают с БД/API АБС напрямую (R3). Разрабатывают Java-специалисты команды АБС.
4. **Ставки ведутся в АБС (F7).** Добавляется новый PL/SQL-модуль ставок и экраны Delphi-клиента с разграничением доступа депозиты/кредиты (F10). АБС — единственный источник истины по ставкам; онлайн-каналы читают кеш из Сервиса депозитов (P1). Разрабатывает команда АБС.
5. **СМС (F5, F8).** Бизнес-логика ОТП-подтверждения и клиентских уведомлений реализуется в модуле СМС Сервиса депозитов, ядро ИБ не дорабатывается (+R2); физическая отправка — через существующий СМС-шлюз телеком-оператора. Разрабатывает команда Сервиса депозитов.
6. **Онлайн-каналы.** Модуль депозитов ИБ — команда ИБ банка (без подрядчика, +R2); страница депозитов и форма заявки на сайте — разработчики сайта (PHP/React.js). Приём заявок с сайта в систему КЦ реализуется через существующий API системы КЦ, доработка минимальна (+R6).
7. **Отказоустойчивость и масштабирование.** Новые сервисы и Kafka разворачиваются кластерно в двух ЦОД и масштабируются горизонтально (R1, P2). Монолит ИБ остаётся в одном ЦОД с ручным переключением на резервный — фиксируется как ограничение (R2 — целевое).

**Ключевые потоки данных:**

- Заявка: ИБ/Сайт → Сервис депозитов → Kafka `deposit-applications` → адаптер АБС → АБС (создание заявки, обработка бэк-офисом).
- Ставки: АБС (модуль ставок) → адаптер АБС → Kafka `deposit-rates` → кеш Сервиса депозитов → REST для сайта и ИБ (P1).
- СМС-события: АБС (статус ставки/депозита) → адаптер АБС → Kafka `deposit-status-events` → Сервис депозитов → СМС-шлюз → клиент.

Доставка организована по семантике «минимум один раз» с идемпотентностью по идентификатору заявки, что закрывает R4 без потери и дублирования заявок.

### **Альтернативы**

1. **Доработка ядра ИБ подрядчиком (депозиты и СМС внутри монолита).** Отклонено: нарушает +R2, создаёт зависимость от подрядчика по срокам и стоимости, не снимает ограничение +R5 (несовместимость с Kafka).
2. **Прямая интеграция ИБ с API/БД АБС (как сделано для платежей).** Отклонено: нарушает R3 и упирается в +R4 — онлайн-поток заявок угрожает работоспособности перегруженной БД АБС.
3. **Ведение ставок в новом сервисе вместо АБС.** Отклонено: F7 прямо требует реализации ведения ставок в АБС, где уже действует процесс согласования ставки и модель доступа (F10).
4. **Сервис депозитов на ASP.NET Core.** Отклонено: экспертиза .NET в банке привязана к платформе подрядчика (Framework 4.5, +R5), тогда как Java-стек уже в проде (система КЦ, Camunda) и по нему есть свои специалисты (S1).
5. **RabbitMQ вместо Kafka.** Отклонено: S2 явно фиксирует выбор Kafka на перспективу.

**Недостатки, ограничения, риски**

- MVP сохраняет ручную обработку заявок бэк-офисом (F6) — сроки открытия депозита не мгновенные; полная автоматизация — цель года.
- Интернет-банк остаётся монолитом в одном ЦОД: требование R2 в MVP закрывается только ручным переключением на резервный ЦОД.
- Kafka — новая для эксплуатации банка технология: требуются обучение команды и настройка мониторинга (S5).
- Повышенная нагрузка на команду АБС (PL/SQL-модуль ставок + интеграционный адаптер) — потенциальное узкое место планирования.
- Кеш ставок вносит задержку публикации изменений (минуты) — допустимо, так как ставки обновляются ежедневно.
- Хранение ОТП-кодов в Сервисе депозитов требует отдельных мер защиты (шифрование в покое, ограниченный TTL).

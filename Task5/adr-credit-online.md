### **Название задачи:** ADR-3. Подача заявки на кредит онлайн (сайт и интернет-банк)

### **Автор:** Иван Щербаков

### **Дата:** 19.07.2026

### **Функциональные требования**

Кредитный процесс с онлайн-подачей заявки реализуется на горизонте года после MVP депозитов. Клиент должен видеть кредитные предложения на сайте и в интернет-банке, подавать заявку онлайн и (для нового клиента с паспортными данными) получать мгновенное решение по скорингу; заявка должна попадать в Кредитный конвейер онлайн, минуя суточный обмен с АБС. Требования — из `Task2/furps.md` (F11–F17, P3, P4, +R3, +R7). Верхнеуровневые Use Cases:

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
| UC1 | Клиент, Сайт, Интернет-банк, Сервис кредитных заявок, Кредитный конвейер | Просмотр кредитных предложений (F11, F15) | Условия кредитов хранятся в Кредитном конвейере → модуль интеграции КК публикует каталог в Kafka → Сервис кредитных заявок кеширует → сайт/ИБ читают REST (P1); в ИБ дополнительно выводится предодобренное предложение клиента. |
| UC2 | Новый клиент, Сайт, Сервис кредитных заявок, Kafka, Кредитный конвейер | Заявка нового клиента с сайта (F11) | Клиент указывает Ф. И. О. и телефон (HTTPS — F9) → Сервис кредитных заявок → Kafka `credit-applications` → модуль интеграции КК создаёт заявку в Кредитном конвейере онлайн, минуя суточный обмен (P3, +R3). |
| UC3 | Новый клиент, Сайт, Сервис кредитных заявок, Система скоринга, БКИ | Мгновенное решение по паспортным данным (F12) | При заполнении паспортных данных Сервис кредитных заявок синхронно вызывает REST API системы скоринга; скоринг использует только данные БКИ (данных АБС о новом клиенте нет) → клиент видит решение на сайте. |
| UC4 | Клиент, Сервис кредитных заявок, СМС-шлюз | СМС о визите в отделение (F13, +R1) | После подачи заявки на кредит Сервис кредитных заявок отправляет клиенту СМС о необходимости прийти в отделение для идентификации. |
| UC5 | Kafka, Интеграционный адаптер АБС, АБС, Менеджер отделения | Продолжение заявки в отделении (F14) | Событие `credit-applications` содержит канонический ID заявки, сгенерированный Сервисом кредитных заявок при подаче → его принимают параллельно модуль интеграции КК (создаёт запись в КК с этим ID) и интеграционный адаптер АБС (из ADR-1, заводит заявку в АБС с тем же ID) → менеджер отделения продолжает работу с той же заявкой по общему ID; для нового клиента без онлайн-заявки создаёт новую. |
| UC6 | Клиент, Интернет-банк, Сервис кредитных заявок, Kafka, Кредитный конвейер, Система скоринга | Заявка в ИБ по предодобренному предложению (F15, F16) | Клиент указывает счёт зачисления и подтверждает СМС-кодом (ОТП как в ADR-1) → заявка через Kafka в КК → повторный упрощённый скоринг, если ситуация изменилась → обработка сотрудником в тот же рабочий день (P3). |
| UC7 | Кредитный конвейер, Kafka, Сервис кредитных заявок, СМС-шлюз, Клиент | СМС о смене статуса заявки (F17) | Модуль интеграции КК публикует смену статуса в Kafka `credit-application-status` → Сервис кредитных заявок отправляет СМС; статус виден в ИБ. |
| UC8 | Кредитный конвейер, Kafka, Интеграционный адаптер АБС, АБС | Зачисление средств при одобрении | Событие «кредит одобрен» → Kafka `credit-approved` → адаптер АБС → АБС проводит зачисление в тот же день (вместо ожидания суточного обмена). |

### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
| R1 | Кредитные онлайн-сервисы работают 24/7 (99,9%); при недоступности скоринга — деградация с ответом «решение придёт по СМС». |
| R3 | Онлайн-каналы не работают с API/БД АБС напрямую — асинхронная развязка через Kafka. |
| R4 | Гарантированная доставка заявок, статусов и СМС-событий кредитного процесса. |
| P1 | Быстрый отклик клиентских операций — кеш каталога предложений и предодобрений в Сервисе кредитных заявок. |
| P2 | Горизонтальное масштабирование Сервиса кредитных заявок и онлайн-каналов. |
| P3 | Заявки по предодобренным предложениям обрабатываются в тот же рабочий день — отдельный онлайн-канал доставки в КК (суточный обмен АБС↔КК ускорить нельзя). |
| P4 | Скоринг не нагружать предрасчётами: пересчёт предодобрений — только по затронутым клиентам и вне пиковых часов. |
| S1 | Стек новых компонентов — Java Spring Boot, PostgreSQL (внутренняя экспертиза). |
| S2 | Kafka как шина асинхронного обмена. |
| S3 | Кредитный функционал онлайн-каналов выносится в отдельный микросервис. |
| S5 | Мониторинг и журналирование онлайн-заявок на кредит (аудит статусов). |
| F10 | Отдельный Сервис кредитных заявок — не смешивать кредитные данные с депозитными. |
| +R1 | Новый клиент проходит идентификацию в отделении. |
| +R3 | Обмен АБС↔КК — по БД раз в сутки, ускорить нельзя (обоснование отдельного онлайн-канала). |
| +R4 | БД АБС перегружена, масштабируется только вертикально. |
| +R7 | Кредитный конвейер (Camunda/Oracle) и скоринг (Python/Flask) — горизонтально масштабируемы 24/7; доработку КК ведёт подрядчик и IT-отдел. |

### **Решение**

**Диаграммы C4** (загружены отдельными файлами в директорию Task5):

- `c4-context-credit.drawio` — уровень 1, контекст. Показывает клиентов и системы кредитного онлайн-процесса: Сайт, Интернет-банк, новый Сервис кредитных заявок, Kafka, Кредитный конвейер (с новым модулем интеграции), Систему скоринга, БКИ, АБС (переиспользуется адаптер из ADR-1) и СМС-шлюз. Новые потоки выделены; существующие (КК↔скоринг↔БКИ, суточный обмен АБС↔КК) показаны как есть.
- `c4-component-credit.drawio` — уровень 2, компоненты Сервиса кредитных заявок и модуля интеграции Кредитного конвейера.

**Логика принятия решений:**

1. **Сервис кредитных заявок — новый микросервис (Java Spring Boot, PostgreSQL, Kafka), зеркальный Сервису депозитов.** Отдельный от него из-за F10 (разделение кредитных и депозитных данных и разграничение владения командами). Функции: каталог кредитных предложений (кеш, P1), приём заявок с сайта и ИБ, ОТП-подтверждение (переиспользование Java-модуля ОТП из ADR-1 как библиотеки), синхронный вызов скоринга для новых клиентов (F12), СМС-уведомления (F13, F17). Разрабатывают Java-специалисты банка.
2. **Модуль интеграции Кредитного конвейера — новые Java-обработчики Camunda** (дорабатывают подрядчик КК и специалисты IT-отдела, +R7): консьюмер заявок из Kafka, публикация каталога условий, статусов и событий одобрения, запуск упрощённого повторного скоринга (F16). Онлайн-канал доставки заявок в КК закрывает P3 в обход суточного обмена (+R3).
3. **Предодобренные предложения (F15).** Формируются из результатов ранее выполненных скорингов, сохранённых в КК; пересчёт — только по затронутым клиентам и вне рабочих часов, без массовых предрасчётов в пик (P4). Публикуются в Kafka → кеш Сервиса кредитных заявок → ИБ.
4. **АБС.** Переиспользуется интеграционный адаптер из ADR-1 с новыми топиками: `credit-applications` для заведения заявки в отделении (UC5) и `credit-approved` для зачисления средств (UC8). Суточный БД-обмен АБС↔КК для остальных данных (клиенты и платежи для скоринга) сохраняется без изменений (+R3).
5. **Канонический (сквозной) ID заявки** генерируется Сервисом кредитных заявок в момент подачи и передаётся в событии `credit-applications`; Кредитный конвейер и АБС создают свои записи с этим же ID (не назначают собственный). Это исключает гонку «кто первым присвоит идентификатор» между параллельными консьюмерами (модуль интеграции КК и адаптер АБС) и обеспечивает связывание и продолжение заявки в отделении (F14) без дублей. Статусы (`credit-application-status`) и одобрение (`credit-approved`) публикуются с тем же ID для идемпотентной обработки (R4).
6. **Границы изменений.** Существующий путь «АБС → КК раз в сутки» для заявок из отделения не меняется; доработки затрагивают только онлайн-канал (учитываем лишь значимое для текущих изменений).

**Ключевые топики Kafka:** `credit-applications`, `credit-offers`, `credit-application-status`, `credit-approved`.

### **Альтернативы**

1. **Онлайн-заявки через АБС по текущему процессу (АБС → суточный обмен → КК).** Отклонено: P3 недостижимо, +R3 запрещает ускорение обмена, +R4 — лишняя нагрузка на перегруженную БД АБС.
2. **Прямые REST-вызовы сайта/ИБ в Кредитный конвейер.** Отклонено: теряется буферизация и гарантированная доставка (R4), пики онлайн-трафика бьют в КК, растёт связность каналов с коробочным продуктом.
3. **Массовый ночной предрасчёт скорингов по всей базе для предодобренных предложений.** Отклонено: нарушает P4.
4. **Расширение Сервиса депозитов кредитным функционалом.** Отклонено: F10 (разделение данных направлений), независимое масштабирование и владение командами.

**Недостатки, ограничения, риски**

- Решение по новому клиенту с сайта опирается только на данные БКИ (F12) — точность ниже, возможны пересмотры решения в отделении.
- Канонический ID заявки требует согласованной поддержки в трёх системах (Сервис кредитных заявок — генерация, КК и АБС — приём с тем же ID) — при нарушении дисциплины возможна рассинхронизация статусов; смягчается идемпотентностью по ID и аудитом (S5).
- Kafka становится критической зависимостью кредитного процесса — требуется кластер в двух ЦОД и мониторинг (S5).
- Синхронный вызов скоринга с сайта в рабочие часы добавляет нагрузку на скоринг (P4 частично компенсируется малым потоком новых клиентов); при недоступности скоринга клиент получает ответ «решение придёт по СМС» (деградация, R1).
- Ядро ИБ не дорабатывается (+R2) — кредитные страницы в ИБ ограничены возможностями модульных доработок платформы.
